# 公众号文章编辑

把视频语言整理成适合手机阅读的公众号文章。默认通过分阶段对话完成，不依赖 n8n、飞书或外部 API。

## 核心约束

- 把当前脚本视为最高优先级事实来源。不得补造人物、数据、经历、工具功能、结果或因果关系。
- 把历史文章和风格样本只用于学习语气、节奏和排版，禁止复用其中的事实、案例、标题、链接或 CTA。
- 保留脚本里的核心对象、场景、痛点、成果和作者有辨识度的表达；删除口头重复和字幕噪声。
- 用户指定修改范围时只修改对应部分，不要借机重写整篇。
- 发现标题、正文或多个脚本版本之间的数据冲突时，明确指出并请用户统一口径。

## 选择模式

用户没有特别说明时，默认使用“视频承接短文”。

1. **视频承接短文**：250–450 字，用痛点、场景和悬念承接视频，不提前讲完结论。
2. **完整图文版**：把脚本的方法、过程和结论整理成可独立阅读的文章。
3. **局部改稿**：保留已有标题、事实、结构和 CTA，只修改用户指出的部分。

只有缺少脚本正文，或模式差异会彻底改变结果且无法推断时，才询问一个简短问题。

## 分阶段编辑流程

不要收到脚本后一次性生成标题、正文、CTA 和排版。按下面三个阶段推进。

### 阶段一：只选标题

读取脚本并在内部确定事实边界，然后：

1. 给出 5 个有真实差异的标题，分别侧重痛点、结果、故事、反常识或目标人群。
2. 标出 1 个推荐标题，并用一句话说明推荐理由。
3. 如果脚本内部出现“定稿标题”，把它视为重要候选，但只有用户在当前请求里明确说“标题已定”时才锁定。
4. 发现数字或结论冲突时，在标题下方指出。
5. 以这句操作提示结束并停止，不要提前写正文：

> 你可以回复“选2”“把3改得更口语一点”，或者直接说“标题定稿：……”；标题确认后我再写正文。

### 阶段二：写正文并询问 CTA

用户选择、修改或明确标题后：

1. 锁定标题，除非用户明确要求，不再改标题。
2. 输出 60–120 字摘要和干净正文。
3. 视频承接模式保留 `[视频号视频插入位]`，暂不追加 CTA，不生成最终排版文件。
4. 按 [references/writing-standard.md](references/writing-standard.md) 静默检查。
5. 以这句操作提示结束；用户也可以同时提出正文修改：

> CTA 你想用自己的，还是使用默认 CTA？可以回复“用默认 CTA”，或者直接粘贴 CTA 文案/发送 CTA 文档链接；正文需要改也可以一起告诉我。

### 阶段三：选择 CTA、追加并排版

用户回复“用默认 CTA”，或提供自己的 CTA 文案/文档链接后进入本阶段：

1. 用户同时提出正文修改时先完成修改，再追加 CTA。
2. “默认 CTA”必须来自当前用户自己的配置，不得把 Skill 作者或其他人的 CTA 当成所有安装者的默认值。
3. 按顺序查找默认 CTA：当前对话明确指定的 CTA、项目中的 `.wechat-article/profile.yaml`、用户目录中的 `~/.config/wechat-article/profile.yaml`。
4. 当前用户尚未配置默认 CTA 时，请用户提供一次 CTA 文案或文档链接，并说明以后可将它设为默认。
5. 读取默认或用户指定的 CTA 文档，并原样追加 CTA 正文、链接和图片。
6. 用户提供 CTA 链接时优先读取最新原文，不依赖缓存副本；无法读取时请用户粘贴内容。
7. 不得擅自把“微信”改成“企业微信”，也不得改动领取条件、预约链接或福利承诺。
8. 当前环境能创建文件时，按 [references/wechat-formatting.md](references/wechat-formatting.md) 生成使用内联 CSS 的 HTML 预览，并插入真实 CTA 图片。
9. 提醒用户在公众号编辑器或草稿箱中检查平台是否重写样式；排版后仍允许继续修改正文或 CTA。

## 标题规则

- 标题中的工具、人物、数字和结果都必须能在当前脚本中找到依据。
- 五个标题要有角度差异，不要只替换同义词。
- 不使用正文无法兑现的夸张承诺。
- 用户说“标题定稿”后锁定标题；用户选择默认或自定义 CTA 后进入追加和排版阶段。

## 正文规则

- 视频承接短文优先采用：真实痛点或具体故事开场—目标读者场景—视频价值与悬念—视频插入位。
- 完整图文版优先采用：场景冲突—问题价值—方法或案例—结论与边界—下一步行动。
- 每段通常 1–3 句，不机械地每句话空一行。
- 少用“首先、其次、最后”和空泛总结，优先使用具体名词、动作、场景和结果。
- 默认不使用 Markdown 粗体符号；排版阶段再处理颜色和强调。

## 结构化调用

只有自动化流程明确要求 JSON 时，才使用 [references/writing-standard.md](references/writing-standard.md) 中的结构。自动化调用可以一次返回全部字段，但必须保留 `stage` 和 `needs_confirmation`，不得绕过事实冲突检查。
