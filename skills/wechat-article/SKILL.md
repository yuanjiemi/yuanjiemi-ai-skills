# 根据视频脚本生成公众号文案 Skill

把视频语言整理成适合手机阅读的公众号文章。默认通过分阶段对话完成，不依赖 n8n、飞书或外部 API。

每次在新对话中首次启用本 Skill 时，响应的第一行必须原样显示：`已启用「根据视频脚本生成公众号文案 Skill」`。即使脚本或 CTA 读取失败，也不得省略、改写或合并这行提示。同一篇文章的后续轮次不要重复提示。这个提示用于让用户确认当前智能体确实加载了 Skill，而不是普通自由写作。

## 核心约束

- 把当前脚本视为最高优先级事实来源。不得补造人物、数据、经历、工具功能、结果或因果关系。
- 把历史文章和风格样本只用于学习语气、节奏和排版，禁止复用其中的事实、案例、标题、链接或 CTA。
- 保留脚本里的核心对象、场景、痛点、成果和作者有辨识度的表达；删除口头重复和字幕噪声。
- 用户指定修改范围时只修改对应部分，不要借机重写整篇。
- 发现标题、正文或多个脚本版本之间的数据冲突时，明确指出并请用户统一口径。

## 选择模式

用户没有特别说明时，默认使用“视频承接短文”。

1. **视频承接短文**：250–450 字，用痛点、场景和悬念承接视频，不提前讲完结论。
2. **完整图文版**：把脚本的方法、过程和结论整理成可独立阅读的文章。
3. **局部改稿**：保留已有标题、事实、结构和 CTA，只修改用户指出的部分。

只有缺少脚本正文，或模式差异会彻底改变结果且无法推断时，才询问一个简短问题。

## 分阶段编辑流程

按下面三个阶段推进。不要在用户选择 CTA 前开始写稿，也不要在用户审核文稿前生成最终排版。

### 阶段一：先选择 CTA

收到脚本后先处理 CTA：

1. 如果用户已经提供 CTA 文案或 CTA 文档链接，直接进入阶段二，不重复询问。
2. 用户明确说“用默认 CTA”时，必须先确认当前智能体能读取默认 CTA；无法读取或尚未配置时，不得假装存在默认 CTA，改为请用户提供一次 CTA 文案或链接并停止。
3. 如果用户没有说明 CTA，先按顺序查找：当前对话明确指定的 CTA、项目中的 `.wechat-article/profile.yaml`、用户目录中的 `~/.config/wechat-article/profile.yaml`。
4. 已找到且可以读取默认 CTA 时，以这句提问结束并停止：

> 这篇文章有你想用的 CTA 吗？有的话可以粘贴文案或发送 CTA 文档链接；没有的话回复“用默认 CTA”。

5. 没有默认 CTA，或当前智能体无权读取默认配置时，以这句提问结束并停止：

> 这篇文章有你想用的 CTA 吗？请粘贴 CTA 文案或发送 CTA 文档链接；如果这次不需要 CTA，也可以回复“本次不加 CTA”。提供后，你还可以把它设为以后使用的默认 CTA。

不要向用户展示本地路径、权限报错或内部读取过程，只需说明当前没有可用的默认 CTA。

### 阶段二：生成完整文稿供审核

确认 CTA 后，一次输出：

1. 5 个有真实差异的标题，分别侧重痛点、结果、故事、反常识或目标人群。
2. 1 个推荐标题及一句推荐理由。正文暂按推荐标题的角度组织，但不要让内容依赖标题才能成立。
3. 60–120 字摘要。
4. 250–450 字视频承接正文，或用户指定模式对应的正文。
5. 视频承接模式中的 `[视频号视频插入位]`。
6. 用户选择的完整 CTA，包括正文、链接和图片说明。
7. 数据或事实冲突提示。

如果脚本内部出现“定稿标题”，把它视为重要候选；只有用户在当前请求里明确说“标题已定”时才只使用该标题。

读取 CTA 文档时使用最新原文，不依赖缓存副本；无法读取时请用户粘贴内容。不得擅自把“微信”改成“企业微信”，也不得改动领取条件、预约链接或福利承诺。

按 [references/writing-standard.md](references/writing-standard.md) 静默检查后，以这句操作提示结束：

> 这版标题、正文和 CTA 可以吗？你可以回复“选2”“把标题3改得更口语”“修改正文第二段”或“更换CTA”；如果内容可以，回复“生成排版预览”。

### 阶段三：修改或生成排版预览

- 用户提出修改时，只修改指定部分，然后再次询问是否生成排版预览。
- 用户回复“生成排版预览”“可以排版”或同义表达时，使用用户选定的标题。用户没有明确选标题但直接批准时，使用阶段二标出的推荐标题。
- 当前环境能创建文件时，按 [references/wechat-formatting.md](references/wechat-formatting.md) 生成使用内联 CSS 的 HTML 预览，并插入真实 CTA 图片。
- 提醒用户在公众号编辑器或草稿箱中检查平台是否重写样式；排版后仍允许继续修改标题、正文或 CTA。

## 标题规则

- 标题中的工具、人物、数字和结果都必须能在当前脚本中找到依据。
- 五个标题要有角度差异，不要只替换同义词。
- 不使用正文无法兑现的夸张承诺。
- 用户选择标题后使用该标题；用户没有明确选择但直接批准排版时使用推荐标题。

## 正文规则

- 视频承接短文优先采用：真实痛点或具体故事开场—目标读者场景—视频价值与悬念—视频插入位。
- 完整图文版优先采用：场景冲突—问题价值—方法或案例—结论与边界—下一步行动。
- 每段通常 1–3 句，不机械地每句话空一行。
- 少用“首先、其次、最后”和空泛总结，优先使用具体名词、动作、场景和结果。
- 默认不使用 Markdown 粗体符号；排版阶段再处理颜色和强调。

## 结构化调用

只有自动化流程明确要求 JSON 时，才使用 [references/writing-standard.md](references/writing-standard.md) 中的结构。自动化调用可以一次返回全部字段，但必须保留 `stage` 和 `needs_confirmation`，不得绕过事实冲突检查。
